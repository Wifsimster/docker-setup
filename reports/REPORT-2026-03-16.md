## Issue 1: SONOFF Mini D Gate Switches — Mutual WiFi Interference

### Affected devices

| Device | Serial | Node | Area | Role |
|--------|--------|------|------|------|
| Ouverture Portail | 25070700037868 | Node:27 | Exterieur | Gate opening |
| Fermeture Portail | 25070700129632 | Node:28 | Exterieur | Gate closing |

Both are **SONOFF WiFi Smart Switch (Mini D)**, firmware v1.0, connected via **Matter over WiFi**, installed in the **same plastic electrical box** outdoors.

### Symptoms

- Randomly, one of the two devices goes offline while the other stays connected.
- The problematic device alternates — swapping the units physically did not fix the issue.
- Node:28 accumulated **240 subscription failures** and **25 offline events** in 72h.
- Node:27 had only **13 failures** but **2 complete offline events** (long duration).
- They were never both stably online for extended periods simultaneously.

### Evidence of mutual interference

Timeline analysis shows correlated behavior:
- On March 15: Node:28 flapped all day (18 "unavailable" events). Node:27 had **zero log entries** — it was silently stable while Node:28 struggled.
- On March 16 at 11:25: Both nodes came back online within **2 seconds** of each other, suggesting they reconnected after the same WiFi recovery event.
- Node:28 then went unavailable at 11:42. Node:27 stayed online.
- Pattern: **one wins, the other loses** — classic near-field co-channel interference.

### Comparison with other Matter nodes

| Node | Device | Failures (72h) | Recoveries | Offline events |
|------|--------|----------------|------------|----------------|
| Node:28 | Fermeture Portail (SONOFF) | 240 | 82 | **25** |
| Node:27 | Ouverture Portail (SONOFF) | 13 | 1 | **2** |
| Node:45 | PC Fixe (Meross) | 124 | 124 | 0 |
| Node:46 | Ecrans (Meross) | 120 | 120 | 0 |
| Node:43 | Docker Server (Meross) | 109 | 109 | 0 |
| Node:44 | NAS (Meross) | 17 | 17 | 0 |
| Node:37 | Freebox (Meross) | 17 | 17 | 0 |
| Node:38 | Firewall (Meross) | 18 | 18 | 0 |
| Node:40 | Switch PoE (Meross) | 22 | 22 | 0 |
| Node:30 | Lumiere TV (Meross) | 22 | 22 | 0 |

Key observation: All Meross plugs have a **100% recovery rate** (failures = recoveries, 0 offline events). Only the two SONOFF gate switches fail to recover and go fully offline. This indicates the SONOFF devices are in a significantly worse network environment, compounded by mutual interference.

### Root cause

Two WiFi devices (ESP32-based, 2.4GHz) operating centimeters apart in the same enclosure cause **near-field co-channel interference**:
- Their antennas desensitize each other's receivers
- Matter's persistent subscription keepalives (every 60s) create collision windows
- The WiFi AP sees both devices at nearly identical signal strength and can't differentiate them well
- Combined with marginal WiFi coverage at the gate (outdoor, distant from AP), one device randomly "wins" the channel while the other gets drowned out

### Recommended fixes

**Fix 1 (best): Physically separate the devices**
Move one SONOFF Mini D to a separate enclosure, even 30-50cm away. RF interference drops with the square of the distance.

**Fix 2: Switch protocol from Matter to MQTT**
Flash the SONOFF Mini D with Tasmota firmware, or use them via eWeLink LAN / MQTT. MQTT is far more tolerant of spotty WiFi:
- No mDNS multicast dependency
- Lightweight TCP keepalives instead of Matter subscriptions
- Graceful reconnection instead of CASE session negotiation

**Fix 3: Improve WiFi coverage at the gate**
Deploy a Unifi outdoor AP (U6-Mesh or similar) closer to the gate area. This benefits all outdoor devices.

**Fix 4: Update firmware**
Both units run firmware v1.0 (initial Matter release). Check eWeLink or SONOFF DIY mode for updates addressing Matter stability.

**Fix 5: Assign static DHCP leases**
Reserve fixed IPs for both devices on your router to avoid DHCP renewal-related drops on weak connections.

## Issue 2: Broadlink RM4 Pro — Permission Error Cycling

### Affected device

| Device | IP | Type |
|--------|-----|------|
| Universal Remote | 192.168.0.14 | Broadlink RM4 Pro |

### Symptoms

The device cycles between Connected and Disconnected every **1-3 minutes**, with `[Errno 13] Permission denied` errors on every disconnection.

**66 error events** in 72 hours.

### Log pattern

```
18:27:26 WARNING Connected to Universal Remote (RM4 pro at 192.168.0.14)
18:28:26 WARNING Disconnected from Universal Remote (RM4 pro at 192.168.0.14)
18:28:26 ERROR   Error fetching data: [Errno 13] Permission denied
18:29:26 WARNING Connected to Universal Remote (RM4 pro at 192.168.0.14)
... repeats every 1-3 minutes
```

### Root cause

`Permission denied` (Errno 13) on Broadlink devices typically means:
- The device is **locked** — another app/controller (e.g., the Broadlink/IHC app on a phone) holds an exclusive session
- The **authentication token has expired** or is invalid
- A firmware update changed the device's auth behavior

### Recommended fixes

1. **Re-authenticate the device** in Home Assistant: remove and re-add the Broadlink integration, go through the pairing process again.
2. **Close competing apps**: Ensure no phone app (Broadlink, IHC) has an active connection to the device — Broadlink RM4 supports limited concurrent sessions.
3. **Lock the device to HA only**: In the Broadlink app, disable cloud control if possible, so HA has exclusive local access.

## Issue 3: ONVIF Camera "Sonnerie" — Network Unreachable

### Affected device

| Device | IP | Type | Integration |
|--------|-----|------|-------------|
| Sonnerie | 192.168.0.101 | ONVIF Camera | onvif (digest auth) |

### Symptoms

Snapshot fetch fails intermittently with `Cannot connect to host 192.168.0.101:80 ssl:default [Network unreachable]`.

**4 errors** observed on March 16 — low frequency but indicates the camera drops off the network periodically.

### Root cause

"Network unreachable" (not "connection refused" or "timeout") means the host is completely off the network at the time of the request. Likely causes:
- Camera power supply issues (outdoor installation)
- WiFi dropout if the camera is wireless
- Camera firmware bug causing periodic reboots

### Recommended fixes

1. **Check if the camera is wired or wireless** — if wireless, same WiFi coverage issue as the gate devices.
2. **Assign a static IP / DHCP reservation** if not already done.
3. **Check camera power** — outdoor cameras can have voltage drops or weather-related issues.
4. **Update camera firmware** — check the manufacturer's site for stability fixes.

## Issue 4: ZHA Zigbee Device — Firmware Version Flip-Flop

### Symptoms

A Zigbee device continuously reports alternating firmware versions every **2-5 minutes**:

```
old_firmware_version='0x00010402' → new_firmware_version='0x00000302'
old_firmware_version='0x00000302' → new_firmware_version='0x00010402'
... repeats indefinitely
```

**77 events** in 72 hours. Additionally, one `zigpy_znp` frame parsing error was observed (truncated ZDO response from device `0x47FD`).

### Root cause

This is not a real firmware update — ZHA is misinterpreting attribute reports from the device. The two "versions" (`0x00010402` and `0x00000302`) likely correspond to two different endpoints or clusters on the same device reporting different values. This is a **known ZHA bug** with certain multi-endpoint Zigbee devices.

### Impact

Low — this is cosmetic log spam. The device itself functions normally. However, it adds unnecessary noise to the logs.

### Recommended fixes

1. **Identify the device**: In HA → ZHA → find the device at Zigbee address `0x47FD` to determine what it is.
2. **Update ZHA/zigpy**: This is a known issue in older ZHA versions — updating Home Assistant may resolve it.
3. **Ignore if not impactful**: This is a warning-level event with no functional effect.

## Additional Notes

### Matter network overall health

The Matter network is generally healthy. The Meross Smart Plug Mini devices (bureau area) show frequent subscription timeouts but **always recover instantly** (100% recovery rate). This is normal Matter behavior for WiFi devices — subscriptions have short liveness timeouts and resubscription is designed to be fast. No action needed for Meross devices.

### Infrastructure observation

The `matter-server` container runs in `network_mode: host`, which is correct and required for mDNS. The Home Assistant container has proper privileged access and device passthrough (`/dev/ttyUSB0` for the Zigbee coordinator). No infrastructure-level changes are needed.
